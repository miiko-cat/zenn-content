# このアプリを作るに至った経緯
私は現在、自動車メーカのAndroidアプリ開発を行っていますが、iOSメンバーと共に開発を推進するに当たり、一つのコードでAndroid、iOS両方にリリースできる(クロスプラットフォーム)Flutterに興味を持つようになりました。

また、臨床心理士によるカウンセリングを受けていた際に、認知行動療法などの心理療法を学んでいた経験と、日常的に生成AIに友達や恋人には話せない悩みを聞いてもらっていました。

以上の経験から、Flutterで両OSにリリースでき、様々なユーザに使ってもらえ、日記とAIを組み合わせることで、ただ日記を投稿するだけではなく、AIにもアドバイスを生成してもらえる疑似カウンセリング的なアプリを作れないかと考えるように至りました。

## Dart
まずは、Flutterで使用されている言語、Dartを学習しました。
変数や条件分岐といった基礎的な文法は、他のプログラミング言語と大きく異なる部分はなかったので、学びやすかったのと、Dart自体がJavaScriptに似ている所があり、以前Webアプリのフロントエンド開発もしたことがあったので、非常に馴染みやすかったです。

### 癖があった箇所
Dartの**Named Parameters（名前付き引数）**については、癖がありました。

Dartの通常の関数は以下の通りに記載できます。
```dart
void main() {
  printName("田中", "太郎");
}

void printName(String lastName, String firstName) {
  print(lastName + firstName);
}
```
このままのコードでもいいのですが、この"田中"という引数が、lastNameの引数であることを明示するのに、名前付き引数を使用します。
```dart
void main() {
  printName(lastName: "田中", firstName: "太郎");
}

void printName({String? lastName, String? firstName}) {
  if (lastName != null && firstName != null) {
    print(lastName + firstName);
  }
}
```
main関数のprintName関数の呼び出し箇所を見てみましょう。lastNameに"田中"、firstNameに"太郎"が引き渡されることが明示されています。

ただし、lastNameとfirstNameはnullを許容する型(`String?`のように後ろに`?`を付ける)でないといけません。それに伴い、Nullチェックまで必要になっています。

これを回避するためには、**`required`を付けて、必須の引数であることを明示**しないといけません。こうすることで、Nullチェックは不要になります。
```dart
void main() {
  printName(lastName: "田中", firstName: "太郎");
}

void printName({required String lastName, required String firstName}) {
  print(lastName + firstName);
}
```

## Flutter
Flutterでチャットアプリや、アニメのキャラクター図鑑、SNSアプリといった実践的なアプリを作成しながら、ウィジェットの扱い方、APIのコール方法、非同期処理などを学びました。

# 技術選定
- モバイルフロントエンド ➡ Flutter
- モバイルバックエンド ➡ Supabase
- サーバレス関数 ➡ SupabaseEdgeFunction
- テキスト分析・テキスト生成 ➡ Gemini
## Flutterを選んだ理由
Flutterは一つのコードで、Android、iOSに両方リリースできること、HotReloadの機能でUIに即座にコード変更を反映でき、開発効率を上げることができることから、Flutterを選択。
## Supabaseを選んだ理由
Supabaseは、バックエンドのSaasで認証などのアカウント管理、APIを介したDBへのデータ登録など、自前で実装しようとすると工数が掛かってしまう部分をカバーでき、DBはFirebaseのようなNoSQLではなく、PostgreSQLなので、今まで学習したRDBの知識を活用できると考えました。
## SupabaseEdgeFunctionを選んだ理由
SupabaseのDBに投稿した日記の内容を登録し、それをGeminiに読み込ませるために、サーバーレス関数であるSupabaseEdgeFunctionを利用しました。

モバイルアプリ(Flutter)からバックエンドの処理を全て呼び出すという手順も考えましたが、Supabaseにデータを登録して、その後GeminiAPIを呼んでとフロント側で制御するのは、コードの設計が難しくなること、GeminiAPIが解析される恐れを危惧し、DB登録⇒AIにデータを読み込ませる一連の処理は、サーバーレス関数で実装することにしました。
## Geminiを選んだ理由
日記の内容を要約、感情の分析、分析した感情と日記の内容からアドバイスを生成してもらうのを生成AIに処理してもらっています。

生成AIはGeminiを使用していますが、その理由としてGeminiは無料枠が存在するので、まずは無料でアプリをリリースすることを試してみたかったからです。

## ユーザのセッション状態で、自動的に画面遷移
Supabaseのauth(認証機能)には、現在使用中のアカウントが、ログイン済みなのか、未ログインなのか状態がStreamで流れてきます。

その機能を利用し、セッション状態を判断し、Home画面を表示するのか、ログイン画面を表示するのかを自動的に切り替えるようにしました。

FlutterにはRiverPodと呼ばれる状態管理をするためのライブラリがあり、そのライブラリで認証状態が変更されたときに、検知できるようにしています。
```dart:auth_providers.dart
// 認証状態監視（StreamProvider）
final authStateProvider = StreamProvider<AuthState>((ref) {
  return ref.watch(authRepositoryProvider).authStateChanges;
});
```
```dart:auth_repository.dart
Stream<AuthState> get authStateChanges => supabase.auth.onAuthStateChange;
```
上記の通り、`authStateChanges`メソッドでSupabaseで管理している認証状態を取し、`StreamProvider`で包むことで、状態がStreamとして流れてくるようにしています。

```dart:app.dart
class AuthGate extends ConsumerWidget {
  const AuthGate({super.key});

  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final authState = ref.watch(authStateProvider);

    return authState.when(
      data: (state) {
        final session = state.session;
        if (session == null) {
          return const AuthPage(mode: AuthMode.login);
        }
        return const HomePage();
      },
      error: (e, _) => Scaffold(body: Center(child: Text('エラー: $e'))),
      loading: () =>
          const Scaffold(body: Center(child: CircularProgressIndicator())),
    );
  }
}
```
アプリの上位ウィジェットで、`authStateProvider`の状態を監視し、そのセッションが存在することがどうかを確認しています。

`final session = state.session;`でセッションの状態を取得し、`if (session == null) { }`でセッションがNullの時にログイン画面、そうでないときはHome画面(HomePage)に遷移するようにしています。

## アプリビルド時に、Supabaseの環境を切り分ける
本プロジェクトでは、バックエンドのSaasとしてSupabaseを利用しているが、開発環境と本番環境でデータを混合させるのを避けるために、別々のプロジェクトを使用しています。

そのため、アプリビルド時に異なる定義ファイルを読み込ませています。

`--dart-define=env={filePath}`という引数を`flutter run`実行時に渡すことで、ローカル環境と開発環境、本番環境で異なるSupabaseのプロジェクトを参照できるようになっています。(ローカル環境は、ローカルに構築したSupabaseを参照する)
```bash
# ローカル環境
flutter run --dart-define=env=dart_defines/.env.local
# 本番環境
flutter run --dart-define=env=dart_defines/.env.prod
# 開発環境
flutter run --dart-define=env=dart_defines/.env.dev
```
実行時の引数で渡したファイルは、`env`ファイルとして読み込まれ、SupabaseのURLやAPIキーを取得できます。
```dart:load_app_config.dart
// Mobile: dotenv
const envFile = String.fromEnvironment('env');
await dotenv.load(fileName: envFile);

return AppConfig(
  supabaseUrl: dotenv.env['SUPABASE_URL']!,
  supabaseKey: dotenv.env['SUPABASE_API_KEY']!,
  sentryDsn: dotenv.env['SENTRY_FLUTTER_DNS']!,
  sentryEnv: dotenv.env['SENTRY_ENVIRONMENT']!,
);
```
ここでは、それぞれ以下の文字列をファイルから取得しています。(SENTRYというエラーログをサーバ上で確認できるライブラリも使用しています)
- SUPABASE_URL
- SUPABASE_API_KEY
- SENTRY_FLUTTER_DNS
- SENTRY_ENVIRONMENT

## Androidでアプリ名とapplicationIdの切り分け

`flavor`という機能を使用することで、Androidアプリはローカル環境と開発環境、本番環境で異なるアプリ名とapplicationIdを付けています。

前述した引数と同じように、`flavor=[dev/prod/local]`を渡すことで切り分けられます。
```bash
# ローカル環境
fluuter run --dart-define=env=dart_defines/.env.local --flavor=local
# 本番環境
fluuter run --dart-define=env=dart_defines/.env.prod --flavor=dev
# 開発環境
fluuter run --dart-define=env=dart_defines/.env.dev --flavor=prod
```
Androidのflavor機能は、**android/app/build.gradle.kts**に定義することができます。
```gradle:build.gradle.kts
productFlavors {
    create("local") {
        dimension = "default"
        resValue(
            type = "string",
            name = "app_name",
            value = "[local]AI分析日記アプリ"
        )
        applicationIdSuffix = ".local"
    }
    create("dev") {
        dimension = "default"
        resValue(
            type = "string",
            name = "app_name",
            value = "[dev]AI分析日記アプリ"
        )
        applicationIdSuffix = ".dev"
    }
    create("prod") {
        dimension = "default"
        resValue(
            type = "string",
            name = "app_name",
            value = "AI分析日記アプリ"
        )
    }
}
```
# 課題点
現状のアプリの課題点は次の通りです。
## MVVM設計が中途半端
コーディングの実装設計として、MVVMを採用しています。

MVVMはView、Model、ViewModelを組み合わせた設計で、Viewは画面、ModelはAPIコールなどのバックエンド処理、ViewModelはその2つを繋ぎ合わせるビジネスロジックを担当という役割に分けて実装するものである。

これはAndroidのJetpackComposeプロジェクトで採用されている設計で、これをFlutterにも適用しています。

現状はログイン、サインアップ画面でこの設計ができているが、他の画面まで反映できていないので、すべての画面に適用する予定です。
## テストコードが不足
テストコードはSupabaseとの通信とのテストを、Mockライブラリを使用してテストできているが、他のテストケースは不足している。

個人開発なので、どこまでテストを実施するのか切り分けができていないが、現状はテストコードが不足しているので、品質を高めるために、テストを充実させるべきである。
## AI分析、アドバイス生成のバックグランド化
日記を投稿した後は、①Supabaseにデータ登録⇒②AIに感情分析⇒③AIにアドバイス生成を一連して実施しているが、②と③はバックグランドで処理しても問題なく、ユーザに操作を中断させてしまう原因になるので、①の処理が終わったタイミングで、メインスレッドに戻すべきである。

AIの処理がまだ終わっていない時は、分析結果は「解析中」などの文言を表示させて、途中経過であることを見せることを検討しています。
## Android,iOSリリース
Flutterはモバイルアプリを主体に使われることを想定しているので、GooglePlay、AppleStoreにもアプリをリリースする予定です。

ただ、GooglePlayはクローズドテストという2週間で15人のテスターに触ってもらう必要があり、AppleStoreは利用料が年間99ドル（日本円で約1.5万円〜程度）になります。

それぞれ公開するハードルが高いので、リリースへ躊躇してしまっていますが、今後対応していきます。
## 課金機能の実装
GeminiのAPIコールは、無料枠だとリクエスト数や、文字数制限が限定されたものになってしまう。
そのため、アプリの利用は1度だけ無料にし、それ以降はサブスクリプション形式で課金してもらい、Geminiの従量課金でもお金の問題が発生しないようにしたい。
